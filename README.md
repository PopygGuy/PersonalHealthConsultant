# Централизованная система мониторинга нормативов для студентов и физической подготовки 

[![Flutter](https://img.shields.io/badge/Flutter-3.x-02569B?logo=flutter&logoColor=white)](https://flutter.dev/)
[![Dart](https://img.shields.io/badge/Dart-3.x-0175C2?logo=dart&logoColor=white)](https://dart.dev/)
[![Python](https://img.shields.io/badge/Python-3.11+-3776AB?logo=python&logoColor=white)](https://www.python.org/)
[![FastAPI](https://img.shields.io/badge/FastAPI-REST-009688?logo=fastapi&logoColor=white)](https://fastapi.tiangolo.com/)
[![SQLite](https://img.shields.io/badge/DB-SQLite-003B57?logo=sqlite&logoColor=white)](https://sqlite.org/)
[![Architecture](https://img.shields.io/badge/Architecture-Client--Server-blueviolet)](#архитектура)

Проект представляет собой клиент-серверную информационную систему для цифровизации процессов в образовательной среде:

- учёт и контроль оценок по нормативам;
- мониторинг физической активности студентов;
- централизованное управление пользователями, факультетами и группами.

Система ориентирована на поэтапное внедрение: от уровня факультета до уровня всего университета.

## Кратко для пользователя

Это единое приложение для трёх ролей:

- студент видит свои оценки и физическую активность;
- преподаватель выставляет и контролирует оценки;
- администратор управляет структурой (факультеты, группы, пользователи).

Главная идея: все данные хранятся централизованно, без дублирования и разрозненных таблиц.

## Цель проекта

Создать единую цифровую платформу, которая:

- упрощает контроль физической подготовки и учебных результатов;
- повышает прозрачность выставления оценок;
- исключает дублирование данных;
- обеспечивает масштабируемость и управляемость.

## Архитектура

Система построена по клиент-серверной архитектуре:

`Flutter (Android/Web) -> REST API (FastAPI/Python) -> Централизованная БД (SQLite, с готовностью к PostgreSQL)`

Почему выбран такой подход:

- нет прямого доступа клиента к БД;
- единая точка контроля и валидации;
- удобное масштабирование;
- поддержка разных устройств с единым UX.

## Роли и доступ

Реализована ролевая модель:

- **Студент**
  - просмотр собственных оценок;
  - просмотр статистики активности;
  - персональные функции профиля и шагомера.
- **Преподаватель**
  - выставление и просмотр оценок;
  - фильтрация студентов по факультетам и группам;
  - мониторинг результатов.
- **Администратор**
  - управление пользователями;
  - управление факультетами, группами, нормативами;
  - сервисные операции восстановления и нормализации справочников.

## Основной функционал

1. **Учёт академических оценок**
   - выставление и обновление оценок по нормативам;
   - хранение истории изменений;
   - удобная фильтрация и поиск.
2. **Мониторинг физической активности**
   - шагомер для студентов;
   - расчёт дистанции и накопительной статистики;
   - персонализация на основе параметров пользователя.
3. **Централизованные справочники**
   - единая БД с факультетами, группами, нормативами;
   - устранение дублей и рассинхронизации.
4. **Адаптивный UI/UX**
   - единый стиль на мобильных и web-экранах;
   - глобальное переключение темы (светлая/тёмная/системная);
   - персонализированные аватары пользователей.

## Технологии и обоснование выбора

- **Flutter (Dart)**
  - единая кодовая база для Android и Web;
  - высокая скорость разработки;
  - адаптивный интерфейс.
- **Python + FastAPI**
  - быстрая реализация REST API;
  - читаемость и удобство сопровождения;
  - понятная интеграция с ORM.
- **SQLAlchemy (ORM)**
  - абстракция от СУБД;
  - снижение количества SQL-ошибок;
  - упрощение миграции SQLite -> PostgreSQL.
- **SQLite (этап пилота)**
  - лёгкий запуск без отдельного сервера;
  - подходит для учебного/факультетского старта.
- **PostgreSQL (этап масштабирования)**
  - целевая СУБД для крупного внедрения;
  - готовность к росту нагрузки и данных.

## Централизованная БД (важно)

В проекте используется единая серверная БД:

- канонический файл SQLite: `backend/phc.db`;
- сущности: `users`, `faculties`, `groups`, `norms`, `grades`, `step_stats`.

Почему это критично:

- нет дублирования и конфликтов;
- единая версия данных для всех ролей;
- простое резервное копирование и поддержка.

### Сервисные скрипты

- Централизация данных из legacy-файла (архивный одноразовый скрипт):
  - `.\backend\.venv\Scripts\python.exe backend/scripts/archive/centralize_db.py`
- Удаление дублей факультетов и перепривязка связей:
  - `.\backend\.venv\Scripts\python.exe backend/scripts/dedupe_faculties.py`
- Заполнение/дополнение справочников:
  - `.\backend\.venv\Scripts\python.exe backend/scripts/seed_initial_catalog.py`
- Создание/обновление администратора `root/root`:
  - `.\backend\.venv\Scripts\python.exe backend/scripts/bootstrap_admin.py`

## Быстрый старт (локальный запуск)

### 1) Backend (FastAPI)

```powershell
cd backend
python -m venv .venv
.\.venv\Scripts\python.exe -m pip install -r requirements.txt
.\.venv\Scripts\python.exe scripts\bootstrap_admin.py
.\.venv\Scripts\python.exe -m uvicorn app.main:app --reload --host 0.0.0.0 --port 8000
```

После запуска API доступен по адресу: `http://127.0.0.1:8000`

### 2) Flutter-клиент

```powershell
cd ..
flutter pub get
flutter run
```

Тестовый вход:

- администратор: `root / root`
- преподаватель: `teacher / 123` (если создан)

## Развертывание API на отдельном ПК (Windows, LAN)

Этот режим нужен, когда backend запускается на выделенном компьютере в локальной сети,
а пользователи мобильного приложения подключаются к нему по IP.

### 1) Подготовка и запуск сервера на ПК

```powershell
cd backend
powershell -ExecutionPolicy Bypass -File .\scripts\setup_server_pc.ps1
```

Скрипт автоматически:

- создаст `backend/.venv` (если нет);
- установит зависимости;
- создаст `.env` из `.env.example` (если нет);
- создаст/обновит администратора `root/root`;
- добавит правило Windows Firewall на порт `8000`;
- покажет LAN-адреса и запустит API на `0.0.0.0:8000`.

### 2) Подключение мобильных клиентов

Собирайте мобильный клиент с уже заданным адресом API (чтобы пользователь вводил только логин/пароль):

```powershell
flutter build apk --dart-define API_BASE_URL=http://192.168.1.42:8000
```

Для запуска в debug:

```powershell
flutter run --dart-define API_BASE_URL=http://192.168.1.42:8000
```

Где `192.168.1.42` — IP выделенного ПК с сервером.

В приложении экран входа больше не требует ввода адреса сервера, пользователь вводит только учетные данные.

Ограничение безопасности: логин `root` разрешен только с устройства, где запущен сервер API.

## Развертывание через VPS (публичный доступ)

Если приложение должно работать не только в локальной сети, backend можно развернуть на VPS с публичным IP.
В этом режиме APK собирается с постоянным адресом API, пользователь устанавливает приложение и входит по выданному логину/паролю.

Подробная инструкция:

- `docs/deployment_vps.md`

Краткий сценарий:

```bash
ssh root@SERVER_IP
cd /opt
git clone https://github.com/PopygGuy/PersonalHealthConsultant.git phc
cd /opt/phc/backend
chmod +x scripts/setup_vps.sh
sudo ./scripts/setup_vps.sh --server-ip SERVER_IP
```

Проверка API:

```text
http://SERVER_IP:8000/docs
```

Сборка APK под VPS:

```powershell
flutter pub get
flutter build apk --release --dart-define API_BASE_URL=http://SERVER_IP:8000
```

Готовый файл для передачи пользователю:

```text
build/app/outputs/flutter-apk/app-release.apk
```

Пользователь устанавливает APK и вводит только логин и пароль. Учетные записи создаются в кабинете администратора.

## Аудит и логи API

В backend включен аудит действий пользователей и запросов API.

### Где лежат логи

- основной аудит-файл: `backend/logs/audit.log`
- ротация файлов: `audit.log.1`, `audit.log.2`, ... (до 5 файлов)
- размер одного файла до ротации: ~2 MB

### Что логируется

- вход в систему: событие `login_success`
- выход из системы: событие `logout_success`
- все запросы `GET/POST/PUT/DELETE`: событие `http_request`
- CRUD-события по ключевым сущностям:
  - `norm_created`, `norm_deleted`, `norm_status_updated`
  - `faculty_created`, `faculty_deleted`
  - `group_created`, `group_deleted`
  - `user_created`, `user_updated`, `user_deleted`
  - `grade_created`, `grade_updated`
  - `steps_saved`

### Примеры строк аудита

```text
2026-05-19 10:20:31 | INFO | AUDIT event=login_success login=teacher role=teacher ip=192.168.1.50
2026-05-19 10:20:31 | INFO | AUDIT event=http_request method=POST path=/token status=200 login=anonymous role=- ip=192.168.1.50 duration_ms=34
2026-05-19 10:21:05 | INFO | AUDIT event=norm_created actor=teacher role=teacher norm_id=... norm_name=Бег 100м is_active=True
2026-05-19 10:22:10 | INFO | AUDIT event=norm_status_updated actor=teacher role=teacher norm_id=... norm_name=Бег 100м is_active=False
2026-05-19 10:24:43 | INFO | AUDIT event=logout_success login=teacher role=teacher ip=192.168.1.50
```

### Быстрый просмотр логов

Windows PowerShell:

```powershell
Get-Content .\backend\logs\audit.log -Tail 100
```

Linux (VPS):

```bash
tail -n 100 /opt/phc/backend/logs/audit.log
```
