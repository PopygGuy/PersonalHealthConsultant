# Диссертационное описание проекта

## Наименование проекта

**Централизованная система мониторинга физической подготовки и академической оценки студентов**  
(англ. **Centralized Student Physical and Academic Monitoring System**)

## Актуальность темы

В образовательной практике контроль физической подготовки и учёт результатов часто ведутся в разрозненных таблицах и локальных инструментах. Это приводит к дублированию информации, усложняет проверку корректности данных, снижает прозрачность выставления оценок и увеличивает нагрузку на преподавателей и администрацию.

Централизованная цифровая система решает указанные проблемы за счёт единой модели данных, ролевого доступа и унифицированного интерфейса для всех участников процесса.

## Объект и предмет исследования

- **Объект исследования:** процессы мониторинга физической подготовки и учёта академических результатов студентов.
- **Предмет исследования:** методы и программные средства построения централизованной информационной системы с разграничением доступа и возможностью масштабирования.

## Цель разработки

Разработать масштабируемую клиент-серверную платформу, которая объединяет:

- учёт оценок по нормативам;
- мониторинг физической активности студентов;
- централизованное управление структурой вуза (пользователи, факультеты, группы, нормативы).

## Основные задачи

1. Реализовать ролевую модель доступа (администратор, преподаватель, студент).
2. Построить REST API для единого взаимодействия клиента и сервера.
3. Обеспечить централизованное хранение данных без дублирования.
4. Реализовать модуль мониторинга физической активности (шагомер и статистика).
5. Обеспечить адаптивный и понятный пользовательский интерфейс.
6. Подготовить систему к масштабированию и миграции на промышленную СУБД.

## Архитектура системы

Система построена по клиент-серверной архитектуре:

`Flutter-клиент (Android/Web) -> REST API (FastAPI/Python) -> Централизованная БД (SQLite/PostgreSQL)`

### Причины выбора архитектуры

- безопасность: клиент не имеет прямого доступа к БД;
- управляемость: сервер централизует бизнес-логику и валидацию;
- масштабируемость: разделение интерфейса и данных;
- переносимость: поддержка разных устройств при единой логике.

## Используемые технологии и обоснование

### Flutter (Dart)

Использован для создания кроссплатформенного интерфейса.

Преимущества:

- единая кодовая база;
- поддержка Android и Web;
- адаптивная верстка;
- высокая скорость разработки.

### Python + FastAPI

Использованы для серверной части и REST API.

Преимущества:

- читаемость и прозрачность кода;
- быстрая разработка и расширение API;
- удобная интеграция с ORM и механизмами авторизации.

### SQLAlchemy (ORM)

Используется как слой абстракции доступа к данным.

Преимущества:

- снижение количества ошибок ручного SQL;
- переносимость между СУБД;
- удобство сопровождения и эволюции схемы.

### SQLite (пилотное внедрение)

Используется как простая стартовая СУБД без отдельного серверного контура.

Преимущества:

- низкий порог внедрения;
- быстрое развёртывание;
- достаточность для пилотных сценариев факультета.

### PostgreSQL (масштабирование)

Рассматривается как целевая СУБД при расширении внедрения.

Преимущества:

- устойчивость к росту данных и нагрузки;
- развитые механизмы администрирования;
- промышленный уровень эксплуатации.

## Ролевая модель и безопасность

### Студент

- просмотр собственных оценок;
- просмотр статистики физической активности;
- персональные настройки профиля.

### Преподаватель

- выставление и корректировка оценок;
- фильтрация студентов по факультетам и группам;
- анализ результатов по нормативам.

### Администратор

- управление пользователями;
- управление факультетами, группами и нормативами;
- запуск сервисных процедур восстановления справочников.

Разграничение прав обеспечивает защиту персональных данных и управляемый доступ к функциям системы.

## Реализованный функционал

1. **Академический контур**
   - выставление оценок по нормативам;
   - хранение истории;
   - фильтрация и поиск по структуре вуза.
2. **Физкультурно-оздоровительный контур**
   - шагомер и накопительная статистика;
   - расчёт дистанции;
   - сохранение персональных параметров.
3. **Административный контур**
   - создание/редактирование/удаление учётных записей;
   - управление факультетами и группами;
   - дедупликация и восстановление справочников.
4. **Пользовательский интерфейс**
   - адаптивные экраны;
   - единая визуальная система;
   - глобальная тема: светлая/тёмная/системная.

## Научная новизна

Научная новизна проекта заключается в интеграции академической оценки и мониторинга физической активности в единой централизованной архитектуре с ролевым разграничением доступа и готовностью к масштабированию.

## Практическая значимость

Решение может быть внедрено поэтапно:

- сначала на одном факультете;
- затем на уровне всего университета.

Это снижает риски внедрения, позволяет получать быстрый эффект и обеспечивает переносимость на другие образовательные учреждения.
